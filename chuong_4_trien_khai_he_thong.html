CHƯƠNG 4: TRIỂN KHAI HỆ THỐNG

Báo cáo thực hành triển khai thực tế, cấu hình vận hành và khắc phục sự cố kỹ thuật - Cửa Hàng 118

Triển khai hệ thống là giai đoạn chuyển dịch sản phẩm từ môi trường lập trình và thử nghiệm cục bộ lên môi trường vận hành thực tế (Production), nhằm cung cấp dịch vụ ổn định, an toàn và sẵn sàng phục vụ người dùng cuối. Đối với hệ thống quản lý cửa hàng dịch vụ công nghệ Cửa Hàng 118, mô hình triển khai hướng tới sự kết hợp linh hoạt giữa các dịch vụ điện toán đám mây hiện đại (Cloud Computing) theo mô hình Micro-hosting và Serverless tích hợp. Chương này sẽ trình bày chi tiết về các môi trường triển khai thực tế, mô tả kiến trúc vật lý kết hợp luồng tích hợp/triển khai liên tục (CI/CD), cung cấp hướng dẫn cấu hình máy trạm phát triển và đi sâu phân tích các bài học khắc phục sự cố kỹ thuật thực tiễn phát sinh trong dự án.

4.1. Môi trường triển khai hệ thống (Deployment Environments)

Hệ thống Cửa Hàng 118 được thiết kế theo cấu trúc tách biệt giao diện (Decoupled Architecture) với các module hoạt động độc lập nhằm tối ưu hiệu năng và khả năng bảo trì. Các thành phần chính bao gồm Website dành cho Khách hàng & Admin, Ứng dụng di động (Mobile App) dành cho Kỹ thuật viên (KTV), Máy chủ xử lý tác vụ dịch vụ (Backend Server), và Hệ quản trị cơ sở dữ liệu quan hệ (Relational Database). Dưới đây là thông số chi tiết của các môi trường vận hành đám mây được lựa chọn:

4.1.1. Môi trường Cloud Hosting Frontend (Vercel)

Ứng dụng giao diện người dùng web (Web Frontend) được xây dựng bằng thư viện React (công cụ đóng gói Vite) và triển khai trên nền tảng Vercel. Đây là nền tảng đám mây được tối ưu hóa chuyên biệt cho các ứng dụng web SPA (Single Page Application) và Jamstack.

Thông số kỹ thuật Chi tiết cấu hình và Vai trò vận hành
Nền tảng triển khai Vercel Cloud Platform (Free Tier - Hobby Plan)
Địa chỉ truy cập (Domain) https://www.cuahang118.online (Đã ánh xạ DNS chính thức)
Cơ chế phân phối Vercel Edge Network (Hệ thống CDN phân tán toàn cầu) giúp tối ưu hóa thời gian phản hồi First Contentful Paint (FCP) dưới 1.2 giây trên mọi khu vực địa lý.
Bảo mật giao thức Tự động cấp phát và gia hạn chứng chỉ SSL/TLS (HTTPS) thông qua tổ chức Let's Encrypt, thực thi mã hóa dữ liệu truyền tải 256-bit.
Tối ưu hóa build Tự động nén mã nguồn (Minification), phân tách mã (Code Splitting), và tối ưu hóa tài nguyên hình ảnh tĩnh khi thực thi quy trình biên dịch sản phẩm.
Bảng 4.1. Thông số môi trường triển khai Frontend trên Vercel

4.1.2. Môi trường Cloud Hosting Backend (Render)

Máy chủ API nghiệp vụ (API Backend) phát triển dựa trên môi trường Node.js (Express Framework) kết hợp thiết lập kết nối thời gian thực qua giao thức Socket.io. Phân hệ này được triển khai trên nền tảng Render theo hình thức dịch vụ ứng dụng web Container-based (Web Service).

Thông số kỹ thuật Chi tiết cấu hình và Vai trò vận hành
Nền tảng triển khai Render Web Service (Free Tier Plan)
Địa chỉ API Endpoint https://cua-hang-118.onrender.com
Môi trường ảo hóa Linux Container (Ubuntu Base Image) chạy trên hạ tầng đám mây được quản lý, tích hợp cài đặt môi trường chạy Node.js phiên bản ổn định (v20.x).
Cơ chế truyền thông Hỗ trợ đầy đủ giao thức HTTP/1.1 và HTTP/2, tích hợp truyền tải luồng thông tin song hướng thời gian thực (WebSockets - WSS) dùng cho Socket.io.
Cơ chế giám sát (Logs) Real-time Application Logging giúp thu thập lỗi runtime, vết luồng API (Morgan middleware) phục vụ công tác giám sát hệ thống liên tục.

4.1.3. Hệ quản trị cơ sở dữ liệu đám mây Supabase PostgreSQL

Cơ sở dữ liệu của dự án sử dụng hệ quản trị cơ sở dữ liệu quan hệ PostgreSQL cấu hình trực tiếp trên hạ tầng đám mây của Supabase. Dịch vụ này mang lại tính ổn định cao và khả năng tương thích toàn diện với Sequelize ORM được dùng trong mã nguồn backend.

Thông số kỹ thuật Chi tiết cấu hình và Vai trò vận hành
Động cơ Database (Engine) PostgreSQL phiên bản 15.x cấp doanh nghiệp.
Đơn vị lưu trữ Cloud Supabase Database (chạy trên hạ tầng ảo hóa AWS - khu vực Singapore để tối ưu hóa độ trễ kết nối mạng về Việt Nam dưới 40ms).
Hình thức kết nối Chuỗi kết nối trực tiếp (Direct Connection) hoặc Connection Pooling thông qua cổng bảo mật 5432 (TCP/SSL mã hóa).
Công cụ quản lý trực quan Supabase Studio Dashboard hỗ trợ thao tác trực tiếp trên bảng, theo dõi tài nguyên phần cứng (CPU, Memory, Disk I/O) và quản lý người dùng SQL chi tiết.
Tính an toàn dữ liệu Hệ thống tự động thực hiện sao lưu trạng thái cơ sở dữ liệu hàng ngày (Daily Auto-backups) và duy trì nhật ký giao dịch cập nhật trực tiếp.

4.1.4. Môi trường phân phối ứng dụng di động (Expo)

Đối với ứng dụng di động dành riêng cho Kỹ thuật viên (KTV) tác nghiệp thực tế ngoài hiện trường, quá trình xây dựng sử dụng framework React Native tích hợp bộ công cụ quản lý Expo. Trong giai đoạn thử nghiệm vận hành nội bộ, việc phân phối được thực hiện linh hoạt thông qua công cụ trung gian Expo Go, giúp rút ngắn thời gian kiểm thử mà không cần đóng gói tệp cài đặt gốc `.apk` hay `.ipa` phức tạp.

  • Môi trường phát triển di động: Expo SDK v54.0 chạy trên nhân React Native v0.81.
  • Phương thức chạy ứng dụng: Thiết bị di động của KTV truy cập ứng dụng thông qua client **Expo Go** tải trực tiếp từ cửa hàng Google Play Store (Android) hoặc Apple App Store (iOS).
  • Cơ chế biên dịch và truyền tải: Khi máy chủ Expo CLI chạy ở chế độ phát triển, nó sẽ sinh ra một liên kết mạng cục bộ/đám mây chứa mã JavaScript bundle đã được biên dịch. Thiết bị di động chỉ cần dùng camera quét mã QR tương ứng để nạp trực tiếp toàn bộ tài nguyên ứng dụng vào bộ nhớ tạm thời của Expo Go, cho phép trải nghiệm đầy đủ các tính năng phần cứng như Camera, Định vị GPS và Thông báo thời gian thực.

4.2. Kiến trúc vật lý và Quy trình CI/CD (System Architecture & CI/CD Pipeline)

Kiến trúc vật lý của hệ thống mô tả cách thức bố trí các thành phần phần mềm trên các máy chủ vật lý/ảo hóa khác nhau và luồng giao tiếp giữa chúng trong quá trình vận hành. Đi kèm với đó, để giảm thiểu các thao tác thủ công dễ xảy ra sai sót, chúng tôi áp dụng quy trình tích hợp liên tục và triển khai tự động hóa (CI/CD) đồng bộ dựa trên kho lưu trữ mã nguồn GitHub.

4.2.1. Sơ đồ kiến trúc vật lý hệ thống

Dưới đây là sơ đồ chi tiết biểu diễn mối tương quan giao tiếp vật lý giữa các thực thể trong hệ thống Cửa Hàng 118:

Hình 4.1. Sơ đồ kiến trúc vật lý hệ thống Cửa Hàng 118

Kiến trúc vật lý này đảm bảo việc chia nhỏ tải trọng xử lý. Khi người dùng truy cập trang web, các tệp tài nguyên tĩnh (HTML, CSS, JS) được nạp tức thời từ nút CDN của Vercel gần người dùng nhất. Các tương tác xử lý nghiệp vụ sau đó sẽ được chuyển tiếp dưới dạng yêu cầu API (HTTPS) hoặc kênh WebSocket (Socket.io) đến máy chủ Render Backend. Máy chủ này thực hiện tính toán nghiệp vụ, xác thực JWT, và thực thi các câu lệnh SQL an toàn thông qua Sequelize ORM kết nối tới cơ sở dữ liệu Supabase PostgreSQL.

4.2.2. Luồng Tích hợp & Triển khai liên tục (CI/CD Pipeline)

Để tối ưu hóa chu kỳ phát triển phần mềm và đảm bảo mã nguồn trên môi trường thực tế luôn đồng bộ với kho lưu trữ chính, dự án thiết lập luồng CI/CD tự động thông qua cơ chế tích hợp Webhooks của **GitHub** với các nhà cung cấp đám mây **Vercel** và **Render**:

  1. Tác nghiệp trên nhánh Git cục bộ: Lập trình viên viết code, sửa lỗi và thực hiện kiểm thử thành công trên máy trạm cá nhân. Tiến hành commit và push mã nguồn lên nhánh chính main của dự án trên GitHub: 
    git add .
git commit -m "feat: trien khai tinh nang tiep nhan bao hanh"
git push origin main
  2. GitHub Webhook kích hoạt luồng triển khai: Ngay khi nhánh main ghi nhận mã xác nhận (commit) mới, hệ thống GitHub sẽ tự động gửi đi tín hiệu Webhook đã được đăng ký trước tới máy chủ điều phối của Vercel và Render.
  3. Triển khai tự động phân hệ Frontend trên Vercel:
    • Vercel kéo (pull) mã nguồn frontend mới nhất về hệ thống build cô lập của mình.
    • Thực thi cài đặt các gói phụ thuộc: npm install.
    • Biên dịch tối ưu hóa tài nguyên tĩnh: npm run build (Vite). Quy trình kiểm tra tính hợp lệ của mã React và Tailwind CSS sẽ được chạy. Nếu phát hiện bất kỳ cảnh báo lỗi cú pháp nào, tiến trình build lập tức dừng lại và báo lỗi về cho nhà phát triển.
    • Nếu build thành công, Vercel cập nhật các tệp tĩnh lên hệ thống Edge CDN toàn cầu với thời gian chuyển giao tức thì (Zero-Downtime).
  4. Triển khai tự động phân hệ Backend trên Render:
    • Render tiếp nhận tín hiệu Webhook, tự động tải xuống mã nguồn backend mới nhất.
    • Khởi tạo một Container ảo hóa mới (Docker-like environment).
    • Thực thi lệnh npm install cài đặt toàn bộ các gói thư viện backend.
    • Chạy lệnh khởi động máy chủ API: npm start (chạy tệp server.js).
    • Render áp dụng cơ chế Blue-Green Deployment: Tiến trình cũ vẫn tiếp tục phục vụ người dùng trong khi tiến trình mới đang được khởi tạo. Render thực hiện giám sát cổng mạng của container mới trong vòng 30 giây. Khi Container mới phản hồi trạng thái hoạt động tốt (Health Check Passed), Render sẽ tự động chuyển hướng toàn bộ các luồng truy cập mới sang Container mới và tiến hành tắt Container cũ một cách an toàn. Quy trình này triệt tiêu hoàn toàn thời gian chết của dịch vụ (Zero-Downtime).

4.3. Hướng dẫn cài đặt và cấu hình phát triển (Installation & Configuration)

Phần này cung cấp tài liệu hướng dẫn kỹ thuật chi tiết dành cho các lập trình viên mới gia nhập dự án để cấu hình, cài đặt và chạy thử nghiệm đầy đủ cả 3 phân hệ (Backend, Frontend, Mobile) của Cửa Hàng 118 trên máy trạm phát triển cục bộ.

Yêu cầu môi trường tiên quyết: Máy trạm phát triển cần được cài đặt sẵn Node.js v18.x hoặc v20.x (LTS), công cụ quản lý mã nguồn Git, trình soạn thảo VS Code, và thiết bị di động thực cài sẵn ứng dụng Expo Go từ chợ ứng dụng.

4.3.1. Cài đặt và cấu hình Phân hệ Backend

Mở ứng dụng Terminal hoặc PowerShell tại thư mục dự án và thực hiện các bước sau:

  1. Di chuyển vào thư mục backend và cài đặt thư viện: 
    cd backend
npm install
  2. Tạo tệp cấu hình biến môi trường cục bộ: Tạo tệp .env tại thư mục gốc /backend và thiết lập thông số kết nối thực tế tới cơ sở dữ liệu Supabase: 
    PORT=3001
# URL kết nối trực tiếp đến Cloud Supabase PostgreSQL
DATABASE_URL=postgresql://postgres.nndixigscjshszmprxda:[mật_khẩu]@aws-0-ap-southeast-1.pooler.supabase.com:6543/postgres?pgbouncer=true
JWT_SECRET=CuaHang118_Sieu_Bao_Mat_Key_2026
# Cấu hình gửi mail thông báo hóa đơn/lịch hẹn tự động
EMAIL_USER=cua-hang-118-noreply@gmail.com
EMAIL_PASS=xxxx_xxxx_xxxx_xxxx # App Password của Gmail
  3. Khởi tạo cơ sở dữ liệu (Database Migrations & Seeders): Thực hiện đồng bộ cấu trúc bảng từ Sequelize Model lên cơ sở dữ liệu đám mây và chèn bộ dữ liệu mẫu mặc định (Tài khoản Admin, linh kiện laptop mẫu, dịch vụ sửa chữa mẫu): 
    npm run seed
  4. Khởi chạy máy chủ API ở chế độ phát triển (Live-reload): 
    npm run dev

    Máy chủ sẽ khởi chạy và lắng nghe kết nối tại cổng http://localhost:3001. Mọi thay đổi trong mã nguồn sẽ tự động được theo dõi và khởi động lại server tức thời thông qua cờ phát hiện node --watch.

4.3.2. Cài đặt và cấu hình Phân hệ Web Frontend

Mở một tab terminal mới và tiến hành thiết lập giao diện Web:

  1. Di chuyển vào thư mục frontend và cài đặt thư viện phụ thuộc: 
    cd frontend
npm install
  2. Cấu hình liên kết API: Tạo tệp .env.local tại thư mục gốc /frontend để cấu hình địa chỉ API Endpoint và liên kết Socket thời gian thực trỏ về máy chủ cục bộ hoặc máy chủ cloud: 
    # Khi chạy thử cục bộ (Local Development)
VITE_API_URL=http://localhost:3001/api
VITE_SOCKET_URL=http://localhost:3001

# Khi trỏ thẳng về máy chủ Production Cloud
# VITE_API_URL=https://cua-hang-118.onrender.com/api
# VITE_SOCKET_URL=https://cua-hang-118.onrender.com
  3. Khởi chạy máy chủ Web phát triển cục bộ: 
    npm run dev

    Vite sẽ biên dịch mã nguồn siêu nhanh và khởi chạy Web Server tại địa chỉ http://localhost:5173. Lập trình viên có thể mở trình duyệt truy cập và trải nghiệm giao diện khách hàng cùng bảng điều khiển quản trị Admin.

  4. Lệnh biên dịch đóng gói sản phẩm phục vụ deploy: 
    npm run build

4.3.3. Cài đặt và cấu hình Phân hệ Mobile App (Kỹ thuật viên)

Ứng dụng di động được kiểm thử nhanh bằng môi trường giả lập hoặc thiết bị di động thực:

  1. Di chuyển vào thư mục mobile và cài đặt các thư viện: 
    cd mobile
npm install
  2. Cấu hình URL kết nối máy chủ API trong mã nguồn: Cập nhật liên kết API trong tệp cấu hình kết nối ứng dụng (như src/config/api.js) trỏ về địa chỉ IP mạng nội bộ của máy chủ phát triển (không sử dụng localhost vì thiết bị di động thực không thể hiểu địa chỉ này): 
    export const API_URL = 'http://192.168.1.15:3001/api'; // Thay thế bằng địa chỉ IP Wi-Fi của máy tính
export const SOCKET_URL = 'http://192.168.1.15:3001';
  3. Khởi chạy máy chủ Expo Metro Bundler: 
    npm start

    Expo CLI sẽ chạy trình đóng gói mã JavaScript và hiển thị một mã QR lớn trực tiếp trên màn hình terminal.

  4. Kết nối và kiểm thử trên thiết bị di động thực:
    • Kết nối máy tính phát triển và thiết bị di động thực vào chung một mạng Wi-Fi nội bộ.
    • Trên điện thoại Android: Mở ứng dụng Expo Go, nhấn vào mục "Scan QR Code" và tiến hành quét mã QR trên màn hình máy tính.
    • Trên điện thoại iOS (iPhone): Mở ứng dụng Camera mặc định của iOS, hướng camera vào mã QR, nhấn vào liên kết mở trong ứng dụng Expo Go xuất hiện trên màn hình.
    • Ứng dụng sẽ tự động tải JavaScript Bundle về máy di động, hiển thị màn hình Đăng nhập của Kỹ thuật viên và sẵn sàng tiếp nhận xử lý đơn hàng sửa chữa thời gian thực.

4.4. Các vấn đề kỹ thuật thực tế gặp phải và Giải pháp khắc phục (Troubleshooting)

Trong quá trình phát triển, tích hợp và triển khai thực tế hệ thống Cửa Hàng 118, đội ngũ phát triển đã đối mặt với nhiều lỗi kỹ thuật đặc thù ảnh hưởng trực tiếp đến trải nghiệm người dùng và tính liên tục của hệ thống. Dưới đây là phân tích chi tiết nguyên nhân gốc rễ và giải pháp khắc phục triệt để cho 3 sự cố kỹ thuật tiêu biểu nhất:

4.4.1. Lỗi mất mát hình ảnh sản phẩm (404 Not Found) do Ephemeral Storage trên Render

Mô tả hiện tượng sự cố: Khi Quản trị viên tiến hành thêm mới linh kiện laptop (như CPU Ryzen, RAM, Ổ cứng SSD) tại trang quản trị Web Admin và tải lên hình ảnh minh họa sản phẩm, hình ảnh hiển thị rõ nét ngay lập tức. Tuy nhiên, sau khoảng 1-2 ngày hoạt động hoặc bất cứ khi nào lập trình viên cập nhật mã nguồn (Deploy bản build mới) lên Render, toàn bộ hình ảnh sản phẩm đã tải lên trước đó đều bị hiển thị lỗi ảnh vỡ (404 Broken Images) trên trang chủ mua sắm trực tuyến.

Phân tích nguyên nhân: Nền tảng Render ở gói dịch vụ miễn phí (Free Tier) sử dụng cơ chế lưu trữ ảo hóa tạm thời gọi là Ephemeral Storage. Khi một container chạy ứng dụng backend được khởi tạo, nó sẽ nhận một phân vùng ổ đĩa ảo độc lập. Bất cứ khi nào hệ thống của Render thực hiện bảo trì định kỳ (Restart container sau 24 giờ không hoạt động để giải phóng RAM) hoặc khi có sự kiện tự động Deploy mã nguồn mới từ GitHub, Render sẽ hủy hoàn toàn container cũ và tạo mới một container sạch từ mã nguồn Git gốc. Do đó, tất cả các tệp hình ảnh sản phẩm lưu cục bộ tại đường dẫn /backend/public/uploads/products/ thông qua API xử lý tải lên (Multer) phát sinh trong quá trình vận hành sẽ bị xóa sạch hoàn toàn, dẫn đến lỗi 404.

Giải pháp khắc phục:

  • Giải pháp ngắn hạn (Đã thực thi trong dự án): Đội ngũ kỹ thuật đã tiến hành gom các hình ảnh sản phẩm linh kiện chuẩn của cửa hàng, đặt cố định vào cấu trúc thư mục tĩnh của dự án cục bộ tại đường dẫn backend/public/uploads/products/, đổi tên khớp chính xác với định danh ghi nhận trong Database mẫu (ví dụ: product-1778775037988-434812152.jpg) và đưa toàn bộ thư mục này vào danh sách Git quản lý (Git Tracking). Khi Render deploy lại, các hình ảnh tĩnh nằm trong mã nguồn Git gốc sẽ luôn luôn được triển khai cố định, đảm bảo hiển thị ổn định 100%.
  • Giải pháp dài hạn (Khuyến nghị nâng cấp): Tách biệt hoàn toàn máy chủ xử lý logic nghiệp vụ và máy chủ lưu trữ tài nguyên đa phương tiện bằng cách tích hợp dịch vụ lưu trữ đám mây ngoài (Cloud Object Storage) thông qua **Supabase Storage** (hoặc AWS S3 / Cloudinary).

Dưới đây là sơ đồ kiến trúc cải tiến kết nối API tải ảnh lên Supabase Storage trực tiếp để giải quyết triệt để lỗi Ephemeral Storage:

4.4.2. Lỗi ẩn nút "Tiếp nhận bảo hành" và Crash logic hiển thị do Invalid Date trên Safari (iOS/macOS)

Mô tả hiện tượng sự cố: Khi Kỹ thuật viên (hoặc Lễ tân) sử dụng máy tính chạy trình duyệt Google Chrome hoặc Microsoft Edge để tra cứu thông tin lịch hẹn và tiến hành tiếp nhận thiết bị bảo hành, toàn bộ giao diện hoạt động chính xác, nút "Tiếp nhận bảo hành" hiển thị đúng điều kiện. Tuy nhiên, khi KTV sử dụng thiết bị di động của Apple (như iPhone chạy trình duyệt Safari mặc định hoặc Chrome trên nhân WebKit của iOS) để thao tác ngoài quầy kỹ thuật, nút bấm này bị ẩn hoàn toàn và ứng dụng liên tục báo lỗi xử lý dữ liệu.

Phân tích nguyên nhân: Nguyên nhân xuất phát từ sự không nhất quán giữa các công cụ xử lý JavaScript (JavaScript Engine) trên các nền tảng trình duyệt khác nhau. Trong khi công cụ V8 (chạy trên Google Chrome và Microsoft Edge) rất linh động trong việc tự động chuyển đổi chuỗi ngày tháng ở định dạng chuẩn ISO hoặc dạng chuỗi thông thường YYYY-MM-DD thành đối tượng Date qua hàm new Date('2026-05-25'), thì công cụ JavaScriptCore của Apple (sử dụng trên trình duyệt Safari) áp dụng tiêu chuẩn nghiêm ngặt và không hỗ trợ cú pháp này, trả về kết quả Invalid Date (giá trị đối tượng không hợp lệ).

Khi ứng dụng thực thi so sánh ngày hẹn bảo hành hoặc ngày đặt hàng khả dụng với ngày hiện tại bằng biểu thức so sánh thời gian, kết quả new Date(item.createdAt).getTime() trên Safari trả về giá trị NaN (Not a Number). Điều này làm sai lệch toàn bộ biểu thức điều kiện logic khiến giao diện hiển thị không hiển thị nút bấm tương ứng.

Giải pháp khắc phục: Xây dựng một hàm xử lý ngày tháng an toàn tuyệt đối (Safe Date Parser) bằng cách sử dụng biểu thức chính quy (Regular Expression) để bóc tách rõ ràng chuỗi ngày tháng trước khi đưa vào hàm khởi tạo. Hàm tiện ích này được đặt trong tệp dùng chung của dự án:

/**
 * Hàm phân tích ngày an toàn tương thích tốt trên Safari/iOS và Android
 * @param {string|Date} dateVal - Chuỗi ngày cần phân tích hoặc đối tượng Date
 * @returns {Date|null} Đối tượng Date hợp lệ hoặc null nếu định dạng lỗi
 */
export const parseSafeDate = (dateVal) => {
  if (!dateVal) return null;
  if (dateVal instanceof Date) return dateVal;

  // Xử lý phân tích chuỗi ngày định dạng chuẩn YYYY-MM-DD hoặc ISO
  const match = dateVal.toString().match(/^(\d{4})-(\d{2})-(\d{2})/);
  if (match) {
    const year = parseInt(match[1], 10);
    const month = parseInt(match[2], 10) - 1; // Tháng trong JS tính từ 0
    const day = parseInt(match[3], 10);
    
    // Nếu có thông tin giờ cụ thể đằng sau
    const timeMatch = dateVal.toString().match(/[T\s](\d{2}):(\d{2}):(\d{2})/);
    if (timeMatch) {
      const hours = parseInt(timeMatch[1], 10);
      const minutes = parseInt(timeMatch[2], 10);
      const seconds = parseInt(timeMatch[3], 10);
      return new Date(year, month, day, hours, minutes, seconds);
    }
    
    return new Date(year, month, day);
  }

  // Trường hợp chuỗi định dạng khác, thử chạy parser mặc định
  const parsed = new Date(dateVal);
  return isNaN(parsed.getTime()) ? null : parsed;
};

Áp dụng hàm helper parseSafeDate để chuẩn hóa giá trị ngày hiển thị trên giao diện Kỹ thuật viên (tệp TechnicianDashboardPage.jsx) và Admin (tệp AdminOrdersPage.jsx) trước khi thực hiện các phép so sánh logic thời gian. Nhờ giải pháp này, nút bấm "Tiếp nhận bảo hành" đã hoạt động ổn định, chính xác trên tất cả các dòng thiết bị di động của Apple chạy iOS cũng như thiết bị Android của KTV tại quầy.

4.4.3. Xử lý lỗi xung đột cổng mạng cục bộ (EADDRINUSE: port already in use 3001)

Mô tả hiện tượng sự cố: Trong môi trường phát triển cục bộ của máy tính chạy hệ điều hành Windows, khi lập trình viên gõ lệnh npm run dev tại thư mục /backend để chạy thử nghiệm mã nguồn, bảng console lập tức hiển thị dòng thông báo lỗi màu đỏ nghiêm trọng: Error: listen EADDRINUSE: address already in use :::3001 và ngay lập tức chấm dứt tiến trình chạy ứng dụng.

Phân tích nguyên nhân: Lỗi EADDRINUSE xảy ra khi tiến trình backend Express cố gắng đăng ký lắng nghe và mở cổng mạng số 3001 trên Card mạng cục bộ, tuy nhiên cổng mạng này đã được hệ điều hành Windows cấp phát cho một tiến trình khác đang hoạt động trước đó. Trong thực tế phát triển, nguyên nhân thường do:

  • Một phiên làm việc trước đó của Node.js bị tắt đột ngột nhưng tiến trình ngầm (Orphan Process) của nó vẫn chạy ẩn trong hệ thống và chưa được giải phóng cổng mạng.
  • Cổng 3001 trùng khớp với cổng mặc định của một số ứng dụng phần mềm nền khác trên Windows (như phần mềm chống virus, dịch vụ đồng bộ đám mây hoặc ứng dụng ảo hóa khác).

Giải pháp khắc phục: Đội ngũ kỹ thuật đã xây dựng quy trình hai bước để giải quyết triệt để vấn đề này trên máy trạm:

  1. Dò tìm và cưỡng chế tắt tiến trình chiếm cổng bằng CMD/PowerShell: Hướng dẫn lập trình viên mở Command Prompt dưới quyền Admin và thực thi chuỗi lệnh kiểm tra ID tiến trình (PID) đang chiếm dụng cổng 3001: 
    netstat -ano | findstr 3001

    Kết quả trả về sẽ hiển thị dòng trạng thái kèm PID ở cột cuối cùng (ví dụ: TCP 0.0.0.0:3001 0.0.0.0:0 LISTENING 4216). Lập trình viên tiến hành cưỡng chế tắt tiến trình chiếm dụng bằng lệnh:

    taskkill /F /PID 4216
  2. Tối ưu hóa mã nguồn Backend để linh hoạt cổng (Port Fallback): Cải tiến tệp server.js của backend để tự động phát hiện biến môi trường hoặc tìm kiếm cổng rảnh dự phòng tự động thay vì gán cứng cổng, kết hợp bổ sung xử lý sự kiện lỗi trực quan: 
    const PORT = process.env.PORT || 3001;
const server = app.listen(PORT, () => {
  console.log(`Máy chủ đang hoạt động an toàn tại cổng: ${PORT}`);
});

// Bắt sự kiện lỗi cổng mạng an toàn
server.on('error', (error) => {
  if (error.code === 'EADDRINUSE') {
    console.error(`Cảnh báo: Cổng mạng ${PORT} đang bị chiếm dụng bởi ứng dụng khác!`);
    console.info('Gợi ý: Hãy chạy lệnh "netstat -ano | findstr ' + PORT + '" để giải phóng cổng hoặc cấu hình cổng khác trong tệp .env.');
    process.exit(1);
  }
});

Giải pháp này giúp đảm bảo sự linh động trong quá trình lập trình thực tế, giúp các thành viên phát triển cục bộ tự tin khởi chạy máy chủ phát triển mà không bị gián đoạn hay bối rối bởi các lỗi xung đột cổng mạng hệ thống.

Tóm lại, thông qua các bước chuẩn bị môi trường cloud chặt chẽ, xây dựng luồng CI/CD tự động hóa thông minh giúp tối đa thời gian uptime và các bài học giải quyết các lỗi thực tế đắt giá, hệ thống quản lý **Cửa Hàng 118** đã được triển khai vận hành thương mại hóa cực kỳ thành công, cung cấp trải nghiệm số hóa mượt mà cho cả Khách hàng, Quản trị viên và Kỹ thuật viên cửa hàng.
